# LLMCLI 导航站：项目说明与项目计划（MVP）

## 1. 项目说明

### 1.1 项目定位
LLMCLI 导航站是一个面向开发者与 AI Agent 工程师的工具目录网站，核心目标是：

- 收集并整理“可通过命令行使用”的工具
- 覆盖两类工具：
  1. 传统软件已有 CLI 能力的工具
  2. 原本没有 CLI，但被包装/桥接后可被命令行调用的工具
- 聚焦“是否适合被 Agent 自动调用”，而不是只做普通软件黄页

一句话定位：
**这是一个“Agent 可调用工具导航站”，帮助用户快速判断某个能力能否被 CLI 化并纳入自动化工作流。**

### 1.2 目标用户
- 个人开发者：想快速找到“某类任务可用的 CLI 工具”
- AI 应用开发者：需要为 Agent/工作流挑选稳定可调用工具
- 自动化工程师：需要将传统工具链升级为脚本化、可编排流程

### 1.3 解决的问题
当前痛点：
- 工具信息分散：GitHub、博客、论坛碎片化
- 不知道工具是否“真的可自动化调用”
- 缺少统一的“Agent 友好度”评估标准

本项目提供：
- 可搜索的工具目录
- 标准化字段（安装方式、输入输出、非交互能力、维护状态）
- 面向 Agent 的实用评分与使用建议

## 2. 价值分析（是否有价值）

### 2.1 有价值，且方向正确
这个项目有现实价值，原因是：

1. AI Agent 与自动化趋势正在放大 CLI 价值
   - 大模型本质上更容易调用命令和脚本，而不是 GUI
2. 传统软件的“CLI 可用性”信息严重不透明
   - 用户经常要自行踩坑验证
3. 市面上缺“面向 Agent 场景”的工具导航
   - 普通工具榜单很多，但缺少“能否稳定被 Agent 调用”的判断层

### 2.2 主要风险
1. 目录站同质化
   - 只“收集链接”容易沦为低价值列表
2. 数据维护成本高
   - 工具更新、停更、接口变化快
3. 早期冷启动困难
   - 没有足够高质量样本时，用户不愿反复访问

### 2.3 怎么做才更有价值（关键建议）
不要做“工具清单”，要做“可执行决策层”。

建议把差异化放在以下三点：

1. Agent 友好度评分（核心）
   - 是否支持非交互模式
   - 是否支持 JSON/结构化输出
   - 是否支持 stdin/stdout 管道
   - 是否有明确退出码与错误语义
2. 可替代方案对比
   - 同类工具横向比较（速度、稳定性、安装成本、学习成本）
3. 最小可复制用例
   - 每个工具提供一个“可直接复制运行”的调用样例

## 3. 项目计划（分阶段）

## 阶段 0：定义标准（1 周）
目标：先统一“什么叫可收录、怎么打分”。

交付物：
- 工具数据 Schema（JSON/YAML）
- Agent 友好度评分规则（v1）
- 收录/剔除标准（例如：6 个月无维护且无替代价值则降权）

## 阶段 1：MVP 上线（2-3 周）
目标：做出可访问、可搜索、可筛选的最小网站。

建议 MVP 功能：
- 工具列表页（分类、标签、搜索）
- 工具详情页（安装、命令示例、评分、维护状态）
- 基础筛选（支持 OS、安装方式、是否 Agent-ready）

内容目标：
- 首批 50 个高质量工具
- 每个工具至少 1 个可运行命令示例

## 阶段 2：质量增强（2 周）
目标：提升“可信度”和“决策价值”。

新增功能：
- 同类工具对比卡片
- 维护状态自动检测（如 GitHub 最近提交时间）
- “推荐场景”与“不推荐场景”字段

## 阶段 3：增长与生态（持续）
目标：让目录变成可持续增长的社区资产。

策略：
- 建立贡献流程（PR 模板 + 自动校验）
- 周更专题（如“最适合 Agent 的 10 个文件处理 CLI”）
- 与社区内容联动（文章、视频、开发者社群）

---

## 4. MVP 范围建议（避免过大）

建议先只做 6 个分类：
- 文件与文本处理
- HTTP/API 调试
- 数据处理与数据库
- 开发与代码辅助
- 系统与运维
- LLM/Agent 工具

建议暂不做：
- 用户账户系统
- 复杂推荐算法
- 在线执行沙箱

先把“收录质量 + 标准化信息 + 快速检索”做好。

---

## 5. 成功指标（前 3 个月）

业务指标：
- 收录工具数：>= 100
- 高质量条目占比（含完整评分 + 示例）：>= 80%
- 周活跃用户（WAU）：持续增长

产品指标：
- 用户完成一次“搜索到可用命令”路径耗时 < 3 分钟
- 工具详情页跳出率逐步下降
- 被收藏/复访工具条目数量上升

---

## 6. 推荐技术路径（务实版）

推荐：**静态站点 + 结构化数据文件**

- 前端：Astro 或 Next.js 静态导出
- 数据源：`data/tools/*.yaml`
- 搜索：Pagefind 或本地索引
- 部署：Cloudflare Pages / Vercel

原因：
- 成本低、上线快
- 易于 PR 贡献
- 后续可平滑扩展到数据库方案

## 8. 方向修正（基于你的新偏好）

你明确偏好的是：`Obsidian CLI`、`Feishu CLI` 这类“应用型 CLI”。

因此建议项目主线调整为：
- 从“通用 CLI 导航”升级为“应用能力 CLI 化导航”
- 优先覆盖协作办公、知识管理、项目管理、沟通平台
- 允许并鼓励“封装 CLI”条目（不是必须官方原生 CLI）

新的差异化重点：
1. 同一个应用能力给出 CLI 方案：官方 CLI / 社区 CLI / MCP 封装 / 自建 wrapper
2. 提供统一命令风格建议，降低用户切换成本
3. 对“封装 CLI”标注成熟度（PoC / 可用 / 生产）

对应候选池见：`docs/应用型CLI优先池50.md`
