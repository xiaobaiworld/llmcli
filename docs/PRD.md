# llmcli 产品需求文档 (PRD)

> 版本：v0.2（Phase 2 重定位）
> 更新时间：2026-04-14

---

## 一、产品定位

**核心问题**：AI agent 要操作飞书、企业微信、Obsidian……能做什么？做不到什么？

**产品回答**：llmcli 是一个**应用 × CLI/MCP 覆盖率目录**。
以具体的生产力应用为中心，精确告诉你：
- 这个应用有哪些 CLI / MCP 工具
- 每个功能是否可以通过命令行完成（覆盖 / 部分覆盖 / 不支持）
- 可用的功能：提供可直接复用的命令示例（输入 → 输出）
- 不可用的功能：说明原因和替代方案

**一句话**：**"给 AI agent 的应用操作手册。"**

---

## 二、目标用户

| 用户 | 场景 |
|------|------|
| 构建 AI agent 的开发者 | 需要知道 agent 能否通过 CLI 自动操作飞书日历、发企业微信消息 |
| 企业自动化工程师 | 评估哪些办公流程可以被 agent 接管，哪些还需要人工 |
| AI 产品经理 | 了解各应用的 agent 可用能力，制定自动化路线图 |

## 三、核心功能需求

### 3.1 首页

- 应用卡片网格（App Grid）
- 每张卡片显示：应用名、覆盖率百分比、支持的工具数量、最后更新时间
- 按覆盖率/类别排序和筛选

### 3.2 应用详情页（/app/[id]）

**区块 1：概览**
- 应用简介
- 可用的 CLI / MCP 工具列表（名称、类型、安装命令、维护方）

**区块 2：功能覆盖矩阵**
- 按业务域（Domain）分组
- 每个功能（Feature）一行：功能名、覆盖状态、所用工具
- 覆盖状态：✅ 支持 / ⚠️ 部分支持 / ❌ 不支持

**区块 3：功能详情**（可展开 / 独立页）
- 覆盖的功能：完整命令示例、真实输入参数、期望输出格式
- 不覆盖的功能：原因说明（GUI-only / API 未开放 / 需要特殊权限）

**区块 4：Gap 总结**
- 不支持功能的整体说明
- 已知的 workaround 或替代方案

### 3.3 搜索

- 全文搜索（Pagefind）：搜"发消息"能找到飞书和企业微信的相关功能
- 按工具类型筛选（CLI / MCP）
- 按覆盖状态筛选

## 四、数据模型（核心）

```
App（应用）
  ├── id, name, description, homepage
  ├── tools[]（该应用可用的 CLI/MCP 工具）
  │   └── name, type(cli/mcp/api), install, repo, maintained
  └── domains[]（业务域，如：消息、文档、日历）
      └── features[]（具体功能）
          ├── id, name, status(supported/partial/unsupported)
          ├── tool（使用哪个工具实现）
          ├── example（命令示例，含输入/输出）
          └── gap_reason（不支持时的原因）
```

## 五、首批覆盖应用（Phase 2 优先级）

| 优先级 | 应用 | 原因 |
|--------|------|------|
| P0 | 飞书/Lark | 官方 CLI 最完整，200+ 命令，中国用户最多 |
| P0 | 企业微信/WeCom | 官方 CLI 刚开源，热度高，7 大域 |
| P1 | Obsidian | 开发者必备，MCP 生态最成熟 |
| P1 | GitHub | gh CLI 最成熟，agent 友好度最高 |
| P2 | Notion | 官方 Remote MCP，无官方 CLI |
| P2 | Linear | 官方 MCP，开发团队常用 |
| P3 | Slack | MCP 认证复杂，agent 友好度低 |

## 六、成功指标（Phase 2 完成标准）

- [ ] 飞书覆盖矩阵：所有主要功能域（≥8个）均有条目，每个有覆盖状态
- [ ] 企业微信覆盖矩阵：7 大域全部有条目
- [ ] 每个覆盖功能：至少 1 个完整命令示例（含输出）
- [ ] 站点可访问（Vercel），搜索可用
- [ ] 移动端基本可用

---

## 七、非功能需求

- **静态部署**：Vercel，零后端
- **数据格式**：YAML（人类可读，易贡献）
- **搜索**：Pagefind（构建时索引，零后端）
- **加载性能**：首屏 < 1.5s（静态资源）
